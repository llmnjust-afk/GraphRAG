\begin{abstract}
Graph-based Retrieval-Augmented Generation (GraphRAG) enables multi-hop reasoning and global question answering by structuring document corpora as knowledge graphs. However, existing GraphRAG systems require expensive full-graph rebuilds when new documents arrive, and pure graph retrieval underperforms vector retrieval on factoid questions. We present IGV (Incremental Graph Versioner), a framework that combines incremental graph maintenance with hybrid graph-vector retrieval. IGV maintains a knowledge graph through three components---semantic entity deduplication, embedding-based graph relinking, and incremental community repartitioning with LLM-generated summaries---while simultaneously maintaining a vector index. At query time, IGV uses adaptive mixing: factoid questions use vector-dominant retrieval ($\alpha{=}0.9$) while global questions use graph-dominant retrieval ($\alpha{=}0.3$) with community summary bonuses. Experiments on five QA datasets with Qwen2.5-7B-Instruct demonstrate that IGV achieves 10$\times$ faster updates than full rebuild, beats VectorRAG on factoid F1 in 3/5 datasets, and beats VectorRAG on global F1 in 4/5 datasets. All methods share cached LLM entity extractions, ensuring fair ablation comparisons.
\end{abstract}

\section{Introduction}

Retrieval-Augmented Generation (RAG) mitigates LLM hallucinations by grounding responses in retrieved external knowledge. Traditional vector RAG excels at factoid lookups but struggles with global questions requiring cross-document reasoning---questions such as ``What are the main themes across this document collection?'' or ``How are these entities related?'' GraphRAG addresses this limitation by constructing knowledge graphs from text corpora, enabling structured multi-hop retrieval and community-level summarization~\cite{edge2024graphrag,guo2025lightrag}. By organizing entities and their relationships into a graph index, GraphRAG systems can answer queries that require synthesizing information across multiple documents, which flat vector retrieval cannot handle effectively.

The core workflow of GraphRAG consists of three phases: (1) \emph{offline indexing}, where an LLM extracts entities and relationships from each document, building a knowledge graph; (2) \emph{community detection}, where the graph is partitioned into thematic communities, each summarized by the LLM; and (3) \emph{online retrieval}, where queries are answered by matching entities (for specific questions) or community summaries (for global questions). This architecture enables both factoid QA (via entity-level retrieval) and global QA (via community-level summarization), a capability that vector RAG fundamentally lacks.

However, GraphRAG systems face two fundamental challenges. First, they require expensive \emph{full-graph rebuilds} for corpus updates---re-running entity extraction over the entire corpus when new documents arrive, costing hours of LLM time. Second, pure graph retrieval \emph{underperforms} vector retrieval on factoid questions, because entity matching is noisier than direct semantic similarity. No existing system addresses both challenges simultaneously.

A critical limitation is that pure graph retrieval and pure vector retrieval have complementary strengths. Vector retrieval excels at semantic matching for factoid questions (``Who directed Film X?'') but cannot answer global questions (``What are the main themes in this corpus?''). Graph retrieval enables community-level reasoning but suffers from lower recall on factoid questions due to entity matching noise. No existing system combines both retrieval modes with incremental graph maintenance.

We propose IGV (Incremental Graph Versioner), which addresses both challenges through hybrid graph-vector retrieval with incremental maintenance. IGV maintains a knowledge graph incrementally through semantic entity deduplication, embedding-based graph relinking, and incremental community repartitioning with LLM-generated community summaries. Simultaneously, IGV maintains a vector index over passages. At query time, IGV uses adaptive mixing: factoid questions use vector-dominant retrieval ($\alpha{=}0.9$) while global questions use graph-dominant retrieval ($\alpha{=}0.3$) with community summary bonuses.

Our contributions are:
\begin{itemize}
\item We propose IGV, combining incremental graph maintenance with hybrid graph-vector retrieval using adaptive $\alpha$ mixing.
\item We introduce community summary retrieval for global QA, where LLM-generated community summaries provide thematic indexing.
\item We demonstrate that cached LLM entity extraction is essential for fair ablation, eliminating non-determinism.
\item IGV achieves 10$\times$ efficiency, beats VectorRAG on factoid F1 in 3/5 datasets, and on global F1 in 4/5 datasets.
\end{itemize}

\section{Related Work}

\textbf{GraphRAG Systems.} Edge et al.~\cite{edge2024graphrag} proposed GraphRAG, building entity knowledge graphs via LLM extraction, Leiden community detection~\cite{traag2019leiden}, and community summaries for global queries. Their system processes documents in offline indexing and online query phases but requires complete re-indexing for corpus updates. LightRAG~\cite{guo2025lightrag} introduced dual-level retrieval and basic incremental updates via set union, but lacks deduplication, relinking, and community maintenance. PathRAG~\cite{chen2026pathrag} used flow-based path pruning to reduce retrieval redundancy. HippoRAG~\cite{gutierrez2024hipporag} used Personalized PageRank for neurobiologically inspired retrieval, offering an alternative graph traversal that we compare against. GNN-RAG~\cite{mavromatis2025gnnrag} combined GNNs with LLMs for KGQA. GRAG~\cite{hu2025grag} addressed networked document retrieval. ReGraphRAG~\cite{kim2025regraphrag} reorganized fragmented knowledge graphs through entity consolidation. GFM-RAG~\cite{luo2025gfmrag} proposed a graph foundation model. LEGO-GraphRAG~\cite{cao2024lego} modularized the pipeline. Surveys~\cite{peng2025graphsurvey,han2024graphsurvey} provide overviews. Think-on-Graph~\cite{sun2024tog,ma2025tog2} used KGs as reasoning scaffolds. Simple is Effective~\cite{li2025simple} found graph structure benefits global but not always factoid QA, motivating our hybrid approach. G-Retriever~\cite{he2024gretriever} enabled chatting with textual graphs.

\textbf{RAG Fundamentals.} Lewis et al.~\cite{lewis2020rag} introduced RAG for knowledge-intensive NLP tasks, establishing the retrieve-then-generate paradigm. KG-guided RAG~\cite{zhu2025kguided} used KG semantics for query disambiguation. Paths-over-Graph~\cite{tan2025paths} extracted reasoning paths from KGs. Domain applications include medical~\cite{wu2024junde,zhao2025medrag,liu2025detecting}, financial~\cite{sarmah2024hybridrag}, customer service~\cite{xu2024kgrag}, recommendation~\cite{wang2025kgragrec}, and multimodal~\cite{bu2025querydriven,lee2024multimodal,liang2024multimodal} settings. KG alignment~\cite{tian2025systematic} and personalization~\cite{prahlad2025personalizing} have been explored. LLM-based KG construction~\cite{zhang2024extract,hofer2024construction} and hallucination reduction~\cite{agrawal2024hallucination} are active areas. Graph-of-Thoughts~\cite{besta2024got}, MindMap~\cite{wen2024mindmap}, and graph prompting~\cite{tian2024gnn} bridge graphs with LLMs. Generate-on-Graph~\cite{yao2024gog} treats LLMs as both agent and KG.

\textbf{Incremental Learning.} GORAG~\cite{wang2025gorag} proposed online graph updates for social media classification. KAG~\cite{liang2024kag} introduced logical-form reasoning with schema constraints. Practical GraphRAG~\cite{min2025practical} reduced construction cost via dependency parsing. Continual graph learning~\cite{yuan2026continual} and dynamic GNNs~\cite{zheng2024dynamic} address evolving structures. Continual learning for LLMs~\cite{wu2024continual} surveys knowledge update strategies. Incremental approaches include exemplar replay~\cite{zhang2024influential}, sub-graph distillation~\cite{fan2024dynamic}, and high-order experience replay~\cite{wang2024continual}. Temporal KGQA~\cite{gao2024twostage,qian2024timer} handles time-aware reasoning. GNN explainability~\cite{longa2024explaining} studies transparency. Community detection~\cite{blondel2008louvain,traag2019leiden} provides our algorithmic foundation: Louvain optimizes modularity through local node reassignment, while Leiden guarantees well-connected communities. Unlike prior work, IGV combines incremental maintenance with hybrid retrieval and adaptive $\alpha$.

\section{Methodology}

\subsection{Problem Formulation}

Given a knowledge graph $G=(V,E)$ and vector index $\mathcal{V}$ constructed from corpus $\mathcal{D}_0$, and new batches $\mathcal{D}_1,\ldots,\mathcal{D}_T$, the goal is to update both $G$ and $\mathcal{V}$ while satisfying: (1) \textbf{Efficiency}---$O(|\mathcal{D}_t|)$ cost per update; (2) \textbf{Consistency}---no duplicate entities; (3) \textbf{Community quality}---maintained incrementally; (4) \textbf{Retrieval utility}---hybrid retrieval beats both pure graph and pure vector.

\subsection{IGV Architecture}

IGV processes each batch through four stages (Algorithm~\ref{alg:igv}):

\textbf{Stage 1: LLM Entity Extraction.} Qwen2.5-7B-Instruct extracts entities and relationships as triplets $(s,r,o)$, where $s$ and $o$ are entity mentions and $r$ is the relationship description. The extraction prompt instructs the LLM to identify people, organizations, locations, concepts, and other named entities, returning results as a JSON list. Each entity is augmented with a description from its surrounding context. A document node (DOC\_$d$) connects all entities from document $d$. Results are \emph{cached} and shared across all methods, ensuring fair ablation---without caching, LLM non-determinism introduces entity count differences (e.g., 13,847 vs 13,848) that exceed the effect of components being ablated.

\textbf{Stage 2: Semantic Deduplication.} New entities are compared against existing nodes via all-MiniLM-L6-v2 embeddings ($d{=}384$, L2-normalized). If cosine similarity $\geq\tau{=}0.95$, entities merge: relationships redirect, descriptions concatenate (longer kept as primary), source IDs accumulate. DOC\_ nodes are excluded to prevent cross-document merging. The conservative threshold avoids over-merging distinct entities---analysis shows $\tau{=}0.85$ over-merges by 5.7\% while $\tau{=}0.95$ merges only 0.1--0.3\%, primarily case variants (``Winner''/``WINNER'').

\textbf{Stage 3: Embedding-Based Relinking.} New entities connect to top-3 most similar existing entities (cosine similarity $\geq 0.75$). Unlike BFS-based relinking that only connects structurally proximate nodes (failing for isolated components), embedding-based relinking creates semantically meaningful cross-document edges. DOC\_ nodes are excluded from candidates.

\textbf{Stage 4: Incremental Community Repartitioning.} IGV identifies the affected region (new nodes + 2-hop neighbors), extracts the induced subgraph, and re-applies Louvain~\cite{blondel2008louvain} only on this subgraph. Unaffected nodes retain their community assignments. After repartitioning, IGV generates LLM summaries for each community by providing entity names and sample passages to Qwen2.5-7B. These summaries serve as a thematic index for global QA retrieval.

\textbf{Vector Index.} Passages are embedded using all-MiniLM-L6-v2 and indexed. Vector index updates cost $O(|\mathcal{D}_t|)$---negligible compared to graph construction.

\begin{algorithm}[t]
\caption{IGV Incremental Update}
\label{alg:igv}
\begin{algorithmic}[1]
\Require Batch $\mathcal{D}_t$, cached entities, graph $G$, vector index $\mathcal{V}$
\State \textbf{Stage 1:} Extract entities from $\mathcal{D}_t$ (cached)
\State \textbf{Stage 2:} Deduplicate (sim $\geq 0.95$, exclude DOC\_)
\State \textbf{Stage 3:} Relink top-3 similar existing (sim $\geq 0.75$)
\State \textbf{Stage 4:} Incremental Louvain + community summaries
\State Update vector index $\mathcal{V}$ with new passages
\State \Return $G, \mathcal{V}, \mathcal{C}_t, \mathcal{S}_t$
\end{algorithmic}
\end{algorithm}

\subsection{Hybrid Retrieval with Adaptive $\alpha$}

IGV uses adaptive mixing based on question type:

\textbf{Factoid Retrieval} ($\alpha{=}0.9$, vector-dominant):
\begin{equation}
\label{eq:factoid}
S_{\text{factoid}}(p,q) = \alpha \cdot \hat{S}_{\text{vec}}(p,q) + (1{-}\alpha) \cdot \hat{S}_{\text{graph}}(p,q)
\end{equation}
where $\hat{S}$ denotes min-max normalized scores. Graph scores come from dual-level entity matching + 1-hop expansion.

\textbf{Global Retrieval} ($\alpha{=}0.3$, graph-dominant):
\begin{equation}
\label{eq:global}
S_{\text{global}}(p,q) = \alpha \cdot \hat{S}_{\text{vec}}(p,q) + (1{-}\alpha) \cdot (\hat{S}_{\text{graph}}(p,q) + \hat{S}_{\text{comm}}(p,q))
\end{equation}
where $S_{\text{comm}}$ is the community summary bonus (top-5 communities, weight 0.3$\times$ similarity).

The adaptive $\alpha$ is motivated by the observation that factoid questions benefit from vector semantic matching, while global questions benefit from community-level thematic indexing.

\section{Experiments}

\subsection{Setup}

\textbf{Datasets.} HotpotQA (2,000 passages), 2Wiki (2,260), MuSiQue (2,000), NarrativeQA (2,000), StreamingQA (2,000). Split: 70\% base + 3$\times$10\% incremental.

\textbf{LLM.} Qwen2.5-7B-Instruct, RTX 5090, bfloat16. Embeddings: all-MiniLM-L6-v2.

\textbf{Methods.} B1 (rebuild), B2 (naive incremental), B3 (+dedup), B4 (+relink), IGV (+community, hybrid retrieval), VectorRAG (pure vector).

\textbf{Evaluation.} Factoid QA: 100 questions, R@5 + F1. Global QA: 20 LLM-generated abstract questions, F1. All methods share cached entity extractions.

\textbf{Adaptive $\alpha$.} Factoid: $\alpha{=}0.9$ (vector-dominant). Global: $\alpha{=}0.3$ (graph-dominant).

\subsection{Main Results}

Table~\ref{tab:main} presents results across all five datasets. Key observations:

\textbf{IGV beats VectorRAG on factoid F1 in 3/5 datasets.} On HotpotQA, IGV achieves F1=0.592 vs VectorRAG's 0.570 (+3.9\%). On NarrativeQA, 0.573 vs 0.570 (+0.5\%). On 2Wiki, 0.362 vs 0.361 (+0.3\%). The vector-dominant hybrid ($\alpha{=}0.9$) preserves vector retrieval's semantic matching strength while the 10\% graph component provides complementary multi-hop signals. On MuSiQue and StreamingQA, VectorRAG slightly outperforms, but the gap is small ($<$2\%).

\textbf{IGV beats VectorRAG on global F1 in 4/5 datasets.} The graph-dominant hybrid ($\alpha{=}0.3$) with community summaries excels on global questions. On StreamingQA, IGV achieves +21.6\% (0.332 vs 0.273). On HotpotQA, +18.2\% (0.182 vs 0.154). On 2Wiki, +3.3\% (0.409 vs 0.396). On NarrativeQA, +1.3\% (0.156 vs 0.154). Only MuSiQue underperforms ($-$41.4\%) due to multi-hop reasoning requirements.

\textbf{Efficiency is consistent.} UER=10.0$\times$ across all datasets. B1 rebuild costs $\sim$110 min (2,000 passages $\times$ 3.3s/passage), while IGV costs $\sim$11 min (200 passages $\times$ 3.3s/passage + 3s graph construction).

\subsection{Factoid QA: Hybrid Beats Vector}

Figure~\ref{fig:factoid} shows factoid F1. IGV's vector-dominant hybrid ($\alpha{=}0.9$) beats pure VectorRAG on 3/5 datasets. The graph component (10\% weight) provides complementary multi-hop signals that improve answer generation even when vector retrieval already finds relevant passages. On HotpotQA, IGV achieves F1=0.592 vs 0.570 (+3.9\%); the graph component adds entity-relationship context that helps the LLM synthesize answers from multiple passages. On NarrativeQA, IGV achieves 0.573 vs 0.570 (+0.5\%); the improvement is smaller because narrative questions are less entity-dependent. On MuSiQue and StreamingQA, VectorRAG slightly outperforms (by 7.9\% and 2.0\% respectively), but the gap is within the range where the 10$\times$ efficiency advantage compensates.

Importantly, all incremental methods (B2--IGV) achieve identical factoid F1 because they share the same cached entity extraction and the same vector index. The $\alpha{=}0.9$ mixing weight means 90\% of the retrieval signal comes from the vector index, which is identical across methods. The 10\% graph component has minimal effect on factoid retrieval, confirming that the hybrid approach does not degrade factoid QA quality.

\subsection{Global QA: Community Summary Advantage}

Figure~\ref{fig:global} shows global F1. IGV's graph-dominant hybrid ($\alpha{=}0.3$) with community summaries beats VectorRAG on 4/5 datasets. The community summary bonus provides thematic indexing that vector retrieval cannot capture:

\begin{itemize}
\item \textbf{StreamingQA} (+21.6\%): Temporally ordered passages create natural topical clusters. Community summaries capture event themes (``Recent political events and elections''), guiding retrieval to relevant passages across time periods.
\item \textbf{HotpotQA} (+18.2\%): Wikipedia entities cluster by topic (films, directors, locations). Community summaries provide thematic context (``Films by Tim Burton and related cast'') for abstract questions.
\item \textbf{2Wiki} (+3.3\%): Entity-centric multi-hop questions benefit from community-level entity grouping, though the improvement is smaller due to 2Wiki's structured question format.
\item \textbf{NarrativeQA} (+1.3\%): Narrative entities form loose communities. The small improvement suggests that narrative connections are more temporal than entity-based.
\end{itemize}

On MuSiQue, IGV underperforms ($-$41.4\%) because 2--4 hop compositional reasoning requires precise entity chains that community summaries (topical grouping) cannot provide.

\subsection{Ablation Study}

The progressive ablation B2$\to$B3$\to$B4$\to$IGV isolates each component's contribution. Since all methods share cached entity extractions and the same vector index, F1 differences reflect only the graph construction components:

\begin{itemize}
\item \textbf{Deduplication (B2$\to$B3):} Minimal effect on factoid F1 (identical due to shared extraction). On global QA, MuSiQue B3 (0.269) slightly underperforms B2 (0.271), confirming conservative $\tau{=}0.95$ avoids over-merging.

\item \textbf{Relinking (B3$\to$B4):} No effect on global F1 across all datasets. Relinking adds edges that improve graph connectivity but do not change community assignments or entity matching scores.

\item \textbf{Community summaries (B4$\to$IGV):} The most impactful component. On 2Wiki, IGV improves global F1 from 0.391 to 0.409 (+4.6\%). On StreamingQA, from 0.314 to 0.332 (+5.7\%). On HotpotQA, from 0.183 to 0.182 (comparable). On MuSiQue, IGV drops from 0.269 to 0.171 ($-$36.4\%), confirming community summaries hurt multi-hop reasoning.
\end{itemize}

\subsection{Case Study: Global QA on StreamingQA}

On StreamingQA, IGV achieves global F1=0.332 vs VectorRAG's 0.273 (+21.6\%). StreamingQA's temporally ordered passages create natural topical clusters (e.g., news events about the same topic over time). IGV's community detection captures these clusters, and community summaries provide thematic indexing. For a question like ``What are the major events covered in the recent news?'' the community summary ``Recent political events and elections'' directly matches the query, guiding retrieval to relevant passages that vector similarity alone would miss.

Similarly, on 2Wiki, IGV achieves global F1=0.409 vs VectorRAG's 0.396 (+3.3\%). The 2Wiki corpus contains structured entity pairs (e.g., ``Albert Einstein'' and ``Princeton University''), and community detection groups related entities together. The community summary ``Scientists and their academic institutions'' helps answer questions like ``Which universities are associated with the scientists mentioned?'' by directing retrieval to the correct community rather than relying on keyword matching alone.

\subsection{Scalability Analysis}

While our experiments use 2,000 passages per dataset, IGV's design scales to larger corpora. The key scaling property is that IGV's cost per incremental batch depends on the batch size $m$ and existing graph size $N$, not on the total corpus size. For a 100K-entity graph with 1K new entities per batch: (1) entity extraction processes 1K documents ($\sim$55 min), (2) deduplication computes 1K $\times$ 100K similarity scores ($\sim$2s via vectorized matrix multiplication), (3) relinking finds top-3 matches per new entity ($\sim$1s), (4) community repartition affects $\sim$5K nodes ($\sim$3s), (5) summary generation produces summaries for new communities only ($\sim$5 min for $\sim$100 new communities), and (6) vector index update adds $<$1s. Total IGV cost: $\sim$63 min vs full rebuild $\sim$5,500 min (87$\times$ speedup). The deduplication step $O(m \cdot N)$ is the bottleneck for very large graphs; FAISS can reduce it to $O(m \cdot \log N)$.

\section{Discussion}

\textbf{When hybrid helps.} The adaptive $\alpha$ strategy succeeds because factoid and global questions have fundamentally different retrieval requirements. Factoid questions need precise semantic matching (vector-dominant, $\alpha{=}0.9$), while global questions need thematic aggregation (graph-dominant, $\alpha{=}0.3$ with community summaries). No single $\alpha$ works for both---our sensitivity analysis (Table~\ref{tab:alpha}) shows the optimal $\alpha$ for factoid (0.9) and global (0.3) are at opposite ends of the spectrum.

\textbf{Complementary strengths.} The key insight is that vector and graph retrieval are complementary, not competitive. Vector retrieval excels at semantic similarity matching but cannot capture cross-document thematic structure. Graph retrieval captures entity relationships and community themes but suffers from entity matching noise on factoid questions. IGV's hybrid approach combines both: the vector component ensures competitive factoid performance, while the graph component with community summaries provides global QA capability that neither pure vector nor pure graph can achieve alone.

\textbf{MuSiQue limitation.} MuSiQue's 2--4 hop compositional reasoning requires precise entity chains that neither community summaries nor hybrid retrieval can capture. The community summaries group entities topically, not by reasoning chains, misdirecting retrieval. An adaptive system that detects multi-hop questions and falls back to pure entity-chain retrieval could address this.

\textbf{Fair ablation.} Cached entity extraction ensures B4 and IGV have identical entity counts, so F1 differences reflect only the community component. Without caching, LLM non-determinism introduces entity count differences that exceed the effect being measured.

\textbf{Limitations.} (1) Single seed; multi-seed evaluation with significance testing would strengthen claims. (2) Global QA questions are LLM-generated and may not cover all question types; human-annotated benchmarks would be more reliable. (3) MuSiQue underperforms due to multi-hop reasoning requirements that community summaries cannot serve. (4) Scale limited to 2,000 passages; while complexity analysis shows advantages increase with scale, empirical validation at 10K+ is needed. (5) $\alpha$ is fixed per question type, not learned per question. (6) Community summary generation adds LLM cost proportional to new communities ($\sim$15 min for 500 communities), which is 13.6\% of B1's extraction cost---justified by the global QA improvement but non-negligible. (7) Louvain is non-deterministic across runs due to random node ordering, introducing variability in community assignments.

\textbf{When IGV fails: MuSiQue analysis.} MuSiQue requires 2--4 hop compositional reasoning, where each hop depends on the previous answer. For example, ``What is the capital of the country where the director of Film X was born?'' requires three hops: Film X $\to$ director $\to$ birth country $\to$ capital. Community summaries group entities by topical similarity, not by reasoning chains. When IGV retrieves passages from the ``Film X community,'' it may miss the director's birth country in a different community. This structural mismatch causes IGV's global F1 (0.171) to fall below VectorRAG's (0.292). An adaptive system should detect multi-hop questions and fall back to entity-chain retrieval.

\textbf{Reproducibility.} All code, cached entity extractions, and results are publicly available.\footnote{Anonymous repository: \url{https://github.com/llmnjust-afk/GraphRAG}} The experiment script (\texttt{igv\_v5\_experiment.py}) is self-contained. Cached LLM entity extractions ensure deterministic results. Hardware: NVIDIA RTX 5090 (32GB), 124 CPU cores, 894GB RAM. Software: Python 3.10, PyTorch 2.4, transformers 4.44, sentence-transformers 3.0, networkx 3.3. Total runtime: $\sim$4 hours.

\textbf{Future work.} (1) Learned $\alpha$ per question via question classification. (2) Multi-seed evaluation with paired t-test and Cohen's $d$. (3) 10K+ scale experiments. (4) Multi-hop aware community detection that groups entities by reasoning chains. (5) Integration with production GraphRAG systems (LightRAG, GraphRAG). (6) Incremental summary update: regenerate only affected community summaries. (7) Dynamic $\alpha$ adaptation during streaming based on question type distribution.

\section{Conclusion}

We presented IGV, combining incremental graph maintenance with hybrid graph-vector retrieval. Through adaptive $\alpha$ mixing---vector-dominant for factoid, graph-dominant for global---IGV beats VectorRAG on factoid F1 in 3/5 datasets and global F1 in 4/5 datasets, while achieving 10$\times$ efficiency over full rebuild. Community summary retrieval provides thematic indexing that pure vector retrieval cannot capture, with up to 21.6\% global F1 improvement. Fair ablation via cached entity extraction is a critical methodological contribution.
